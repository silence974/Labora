# 系统架构设计

## 设计理念

Labora 是一个专注于**科研文献研究**的 AI 助手桌面应用，核心能力：
- 文献检索、阅读、分析
- 知识提取、关系构建
- 研究问题探索、综述生成

## 部署模式

### 主要模式：桌面应用（推荐）
- **Electron 容器** + **React 前端** + **Python 后端**
- 使用 **Pretext** 进行论文文本排版和布局
- 一键安装，无需用户配置 Python 或 Node.js 环境
- 跨平台支持：Windows / macOS / Linux

### 可选模式：CLI / Web
- **CLI 模式**: 命令行工具，适合开发者和自动化场景
- **Web 模式**: 浏览器访问，适合团队协作（可选）

两种模式共享相同的核心逻辑层（Python 后端）。

## 各层职责

### 1. Electron 桌面容器层
- **进程管理**: 启动和管理 Python 后端进程
- **窗口管理**: 创建和控制应用窗口
- **生命周期**: 处理应用启动、退出、更新
- **安全隔离**: contextIsolation 确保安全性
- **资源加载**: 加载前端静态资源

### 2. 前端层 (React + Vite + Pretext)
- **PaperReader**: 论文阅读器
  - 使用 **Pretext** 进行文本布局和排版
  - PDF 渲染和标注
  - 阅读进度管理
- **ResearchBoard**: 研究看板
  - 任务状态可视化
  - 进度追踪
- **KnowledgeGraph**: 知识图谱可视化
  - D3.js 图谱渲染
  - 交互式探索
- **Library**: 文献库管理
  - 文献列表和搜索
  - 标签和分类

### 3. Python 后端层 (FastAPI)
- **API 服务**: 提供 RESTful API
- **Agent 调度**: 执行 LangGraph 工作流
- **工具调用**: 调用学术搜索、PDF 解析等工具
- **记忆管理**: 读写短期和长期记忆
- **打包部署**: PyInstaller 打包为独立可执行文件

### 4. 核心业务层
- **ResearchEngine**: 研究任务的核心逻辑
- **ChatEngine**: 对话管理和上下文维护
- **KnowledgeEngine**: 知识库的增删改查
- **统一接口**: 所有接口层都调用这一层

### 4. Agent 编排层
- **LangGraph Workflow**: 定义研究工作流
- **节点执行器**: Planner、Researcher、Analyzer、Synthesizer、Evaluator
- **工具管理**: 统一管理外部工具调用
- **记忆接口**: 读写短期和长期记忆

### 4. 记忆层
- **Redis**: 会话缓存、任务队列、实时状态
- **PostgreSQL + pgvector**: 持久化存储、向量检索
- **Neo4j**: 知识图谱、实体关系（可选）

## 通信流程

### 桌面应用通信流程

```
用户操作 (点击、输入)
    ↓
React 前端组件
    ↓
API 调用 (fetch/axios)
    ↓
HTTP Request → http://127.0.0.1:8765/api/...
    ↓
Python FastAPI 接收
    ↓
调用核心业务层 (ResearchEngine/ChatEngine)
    ↓
执行 LangGraph 工作流
    ↓
调用工具 (Arxiv/PDF Parser/LLM)
    ↓
读写记忆层 (Redis/PostgreSQL)
    ↓
返回结果
    ↓
HTTP Response (JSON)
    ↓
React 前端更新 UI
    ↓
Pretext 渲染文本内容
    ↓
用户看到结果
```

### Electron 启动流程

```
用户启动应用
    ↓
Electron 主进程启动
    ↓
启动 Python 后端进程
    ↓
等待 Python 服务就绪 (轮询 /health)
    ↓
创建 BrowserWindow
    ↓
加载前端页面 (index.html)
    ↓
前端初始化，获取 API URL
    ↓
应用就绪，用户可操作
    ↓
用户关闭应用
    ↓
Electron 捕获退出事件
    ↓
终止 Python 进程
    ↓
清理资源
    ↓
应用退出
```

## 技术选型权衡

### 桌面应用 vs Web 应用

**选择桌面应用的原因**:
- ✅ 本地运行，数据隐私安全
- ✅ 无需服务器部署和维护
- ✅ 更好的性能（本地 LLM 调用）
- ✅ 离线可用
- ✅ 更好的文件系统访问（PDF 管理）
- ✅ 原生体验（窗口管理、系统集成）

**Web 应用的优势**（可选模式）:
- 团队协作和共享
- 跨设备访问
- 无需安装

### Electron vs Tauri

**选择 Electron**:
- ✅ 成熟稳定，生态丰富
- ✅ 跨平台支持完善
- ✅ 开发体验好
- ✅ 社区资源多

**Tauri 的优势**:
- 更小的安装包
- 更低的内存占用
- 但生态相对不成熟

### PyInstaller vs Nuitka

**选择 PyInstaller**:
- ✅ 简单易用
- ✅ 支持大多数 Python 库
- ✅ 社区成熟

**Nuitka 的优势**:
- 更好的性能（编译为 C）
- 但配置复杂

### Pretext 文本布局

**为什么使用 Pretext**:
- ✅ 高质量文本排版
- ✅ 精确的文本测量和换行控制
- ✅ 适合学术论文的复杂排版需求
- ✅ 支持多列布局、脚注等学术特性

**使用场景**:
- 论文阅读器的文本渲染
- PDF 内容的重新排版
- 笔记和标注的布局

## 扩展性设计

### 水平扩展
- FastAPI 服务无状态，可多实例部署
- Redis 使用集群模式
- PostgreSQL 读写分离

### 垂直扩展
- 增加 GPU 资源用于向量计算
- 增加内存用于缓存热数据
- SSD 存储提升数据库性能

## CLI 接口设计

### 命令结构
```bash
labora [全局选项] <命令> [命令选项] [参数]
```

### 核心命令
- `labora chat` - 交互式对话
- `labora research <问题>` - 执行研究任务
- `labora read <论文>` - 阅读论文
- `labora search <关键词>` - 搜索论文
- `labora library` - 文献库管理
- `labora serve` - 启动 Web 服务

### 配置文件
位置：`~/.labora/config.yaml`

```yaml
llm:
  provider: openai
  model: gpt-4
  api_key: ${OPENAI_API_KEY}

database:
  url: postgresql://localhost:5432/labora
  redis_url: redis://localhost:6379/0

agent:
  max_iterations: 3
  default_tools: [arxiv, semantic_scholar, pdf_parser]
```

## API 接口设计

### REST API
- `POST /api/research` - 创建研究任务
- `GET /api/research/{task_id}` - 获取任务状态
- `POST /api/papers/search` - 搜索论文
- `GET /api/papers/{paper_id}` - 获取论文详情
- `POST /api/notes` - 添加笔记

### WebSocket
- `/ws` - 实时推送任务进度和结果

## 安全性考虑

- API 认证：JWT Token
- 数据加密：敏感数据加密存储
- 访问控制：基于角色的权限管理
- 审计日志：记录所有关键操作
